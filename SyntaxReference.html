ACDL Language Reference

This is a complete reference for the Agentic Context Description Language (ACDL). The language declaratively specifies prompt structures sent to large language models, separating structural concerns from content and implementation.

Specification Structure

An ACDL specification defines a named prompt template parameterized by optional indices. The body consists of an ordered sequence of prompt blocks—role messages, label blocks, and control flow constructs—which may be freely interleaved:

PromptName[idx1, idx2, ...]: {
    <prompt-blocks>
}

A single file may contain multiple specifications, each defining a separate prompt template. This is useful when prompts reference each other via label blocks:

PromptName1[idx1, idx2, ...]: {
    <prompt-blocks>
}

PromptName2[idx1, idx2, ...]: {
    <prompt-blocks>
}

Role Messages

Every message carries exactly one role. Four roles serve the chat format; a fifth pseudo-role serves the legacy completion format:

MarkerPurpose
S:System—instructions, persona, tool descriptions, behavioral constraints
U:User—external input, observations, tool results
A:Assistant—prior model outputs, reasoning traces, chosen actions
T:Tool—structured tool call results
N:None—single unstructured text block (completion format only)

Role messages support two syntactic forms. The multi-line form encloses content in braces and permits any combination of content elements and control flow:

S: {
    INSTRUCTIONS
    AVAILABLE_TOOLS
    env.datetime[@t]
}

The single-line form omits braces and accepts exactly one content element—a context variable, template, or function call:

U: env.user_question[@t]
A: resp.answer[@t]
S: INSTRUCTIONS

Control flow constructs (ForEach, If, Switch) are not permitted in single-line role messages. To include control flow inside a role message, the multi-line braced form must be used.

Completion Format: The N: role imposes two additional constraints: (1) exactly one N: block may appear per prompt, and (2) no chat roles may appear in the same specification.

Scoping Rules

The language enforces a strict two-level scope that mirrors the structure of LLM chat APIs:

  • Top level (prompt body, outside any role block): only role messages, label and marker blocks, control flow constructs, name definitions, and comments are permitted
  • Inside role blocks: valid elements are context variables, functions, templates, control flow, comments, name definitions, marking blocks, and break/continue

Role messages may not appear inside other role messages.

Context Variables

Context variables reference dynamic runtime data. The general syntax is:

namespace.path[indices]

where namespace is one of four reserved prefixes:

NamespaceUse Cases
envEnvironment—external inputs, observations, sensor readings, user queries, game state
sysSystem—agent state, memory contents, tool configurations, action histories
respResponse—prior LLM outputs, reasoning traces
promptPrompt—content from labeled sections of this or other prompts

Paths may be nested using dot notation to reach into sub-fields of structured data. Indices may appear at any level of the path:

env.user_question[@T]           // user question at time T
sys.agent_desc                   // constant agent description
sys.tool[@t].tool_response      // tool response at time t
prompt.History[@T-1]            // history label at previous step

Indices

Indices address specific elements along one or more dimensions. Two types are distinguished syntactically.

Time Indices

The special symbol @ denotes the primary time dimension. The current time is denoted with capital letters: @T for the main time step, and I, J, etc. for sub-steps. When iterating, the corresponding lower-case letters are used.

@T      // Current time step
@T-1    // Previous time step
@0      // Initial step
@T.I    // Current substep of current turn
@t.i    // Substep i of turn t (in loops)

Examples

// Iterating over all previous turns
ForEach(@t: range(1, @T-1)) {
    env.observation[@t]
}

// Nested: substeps within each previous turn
ForEach(@t: range(1, @T-1)) {
    ForEach(i: range(0, @t.substeps)) {
        sys.action[@t.i]
    }
}

Non-Time Indices

Non-time indices have no prefix and address other dimensions—named entities or context-variable-valued keys:

[sys.agent_name]
[bomb]

Multiple indices are comma-separated: env.bomb_location[@t, bomb]. Standard arithmetic operators (+, -, *, /, %) are permitted in all index positions.

Templates

Templates are ALL_CAPS placeholders representing text blocks whose content is specified at instantiation time. They describe the semantic purpose of a text section without fixing its wording.

INSTRUCTIONS           // Task explanation
AVAILABLE_TOOLS        // Tool list
QUERY(sys.agent_name) // Parameterized template

Templates may accept arguments in parentheses, enabling parameterized text.

Functions

Functions represent computed content—summarization, retrieval, formatting, or any transformation. They are declared by name and purpose without defining their implementation; the name conveys semantic intent.

The syntax is functionName(arg1, arg2, ...)[indices]. Function names use camelCase (distinguishing them from ALL_CAPS templates).

summarize(prompt.History[@t])
k_relevant_docs(env.user_input[@T])
range(1, @T-1, 2)

The built-in range(start, stop, step) function generates numeric sequences for use in ForEach loops. The step argument is optional and defaults to 1. The range is inclusive.

Control Flow

Three constructs govern dynamic prompt structure. All three may appear both at the top level (producing or gating entire role messages) and inside role blocks (controlling content within a single message).

ForEach

Iterates over ranges or collections to produce repeated structures:

ForEach(variable: iterable) {
    <body>
}
// Top-level: produces role messages
ForEach(@t: range(1, @T-1)) {
    U: env.user_question[@t]
    A: resp.answer[@t]
}

// Inside a role: produces content
U: {
    ForEach(bomb: env.bombs) {
        env.bomb_location[@t, bomb]
    }
}

If / ElseIf / Else

Conditionally includes or excludes blocks based on runtime state:

If (@T > 1) {
    ForEach(@t: range(1, @T-1)) {
        U: env.user_input[@t]
        A: resp.answer[@t]
    }
}

If (sys.tool[@t] == get_clarification) {
    U: env.user_input[@t]
}
ElseIf (sys.tool[@t] == search) {
    A: env.search_results[@t]
}
Else {
    A: sys.tool_used[@t].tool_response
}

Switch / Case / Default

Selects among multiple alternatives based on the value of an expression:

Switch (sys.action_type[@t]) {
    Case "search" {
        U: env.search_results[@t]
    }
    Case "calculate" {
        U: env.calculation[@t]
    }
    Default {
        U: env.fallback[@t]
    }
}

The break and continue keywords are available inside loops with their standard semantics.

Early Termination

The PromptEndsHere when construct signals that, if the given condition is true, the prompt ends at that point—no further content is appended to the message sequence sent to the LLM for that turn.

PromptEndsHere when (@T == @t && @T.0)

Label Blocks

A label block groups a sequence of prompt blocks under a name, making that section addressable for later reference:

Prompt[@t]: {
    Setup {
        S: INSTRUCTIONS
        U: env.initial_context[@0]
    }
    History {
        ForEach(@t: range(1, @T-1)) {
            U: env.user_question[@t]
            A: resp.answer[@t]
        }
    }
}

Labels enable two composition patterns:

  • Cross-time references: prompt.History[@t-100] accesses a labeled section from a prior time step
  • Cross-prompt references: prompt.Main.Reasoning[@t-1] accesses a labeled section in a different specification

Mark Blocks

A mark block annotates a section of the specification for visual emphasis in the rendered output. It places a bracket (]) along the side of the marked content, with a number identifier displayed beside it.

MARK 1 {
    <prompt-blocks>
}

Marks are purely presentational—they do not affect prompt semantics or scoping.

Name Definitions

A name definition binds a symbolic name to an expression, allowing complex expressions to be written once and referenced concisely:

Name docs := k_relevant_docs(env.user_input[@T])
ForEach(i: range(1, $docs.len)) {
    $docs[i].source
    $docs[i].content
}

Once defined, the name is referenced using the $ prefix: $var_name.

List Comprehensions

Name definitions also support list comprehensions:

Name relevant_summaries := [sys.summary[@t] for t in range(@T, @T-900, 100)]
compress_summaries($relevant_summaries)

Comments

Comments use // syntax and may appear on any line. An inline comment, placed after a content element, renders alongside that element. A standalone comment renders at the current level of nesting:

S: {
    // This comment appears on its own line
    INSTRUCTIONS   // Renders beside INSTRUCTIONS
    AVAILABLE_TOOLS
}
